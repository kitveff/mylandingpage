# CLAUDE.md — Web Development Project

> Этот файл управляет стратегией работы: когда работать одному, когда запускать агентов, как экономить токены без потери качества.

---

## 🧠 Философия проекта

- Каждый сайт — уникальный продукт, а не шаблон.
- Сначала думаем, потом пишем код.
- Агенты — инструмент, а не цель. Используй только когда дают реальную пользу.
- Безопасность и качество важнее скорости.
- Дизайн — не украшение, а функция.

---

## 🎯 Первый шаг — оценка задачи

**Перед любой работой** проведи оценку по этой таблице. Выбери режим и работай в нём.

| Критерий | Solo | Duo | Team |
|---|---|---|---|
| Страниц | 1–2 | 3–6 | 7+ |
| Компонентов | < 8 | 8–20 | 20+ |
| Есть бэкенд / API | нет | возможно | да |
| Есть аутентификация | нет | нет | да |
| Уникальная дизайн-система | нет | да | да |
| Несколько разработчиков | нет | нет | да |
| Срок > 2 недель | нет | возможно | да |

**Правило:** если 3+ критерия указывают на более высокий режим — переходи в него.

**Объяви режим пользователю** перед стартом:
> "Режим: Solo / Duo / Team — потому что [конкретная причина]"

## ⚡ SOLO MODE — один контекст, без агентов

**Когда:** лендинг, промо-страница, визитка, правка существующего проекта, быстрый прототип.

**Экономия токенов:** максимальная. Весь проект — один разговор.

### Порядок работы

```
1. Оцени задачу (таблица выше)
2. Беглый анализ Dribbble — 5 минут, 5-7 референсов, фиксируй в памяти
3. Определи стек — выбери очевидное, не думай долго
4. Пиши код — mobile-first, семантика, CSS-переменные
5. Playwright MCP — один smoke-прогон в конце
6. Пройди встроенный security-чеклист
```

### Встроенный security-чеклист (Solo)
- [ ] Нет API-ключей в коде
- [ ] Формы валидируются (хотя бы на клиенте)
- [ ] HTTPS-ссылки везде
- [ ] `.env` не попадёт в репозиторий

### Playwright — минимальный smoke (Solo)
```javascript
// tests/smoke.spec.js
test('страница загружается, нет console errors', ...)
test('навигация работает', ...)
test('мобильная версия 375px корректна', ...)
test('формы валидируются', ...) // только если есть формы
```

## 🔀 DUO MODE — два специализированных агента

**Когда:** сайт 3–6 страниц, нужен осознанный дизайн, есть компонентная система, планируется итерация стилей через 21st.dev.

**Агенты:**

### Агент A — `design-dev`
Отвечает за дизайн-исследование и весь код.

```
Инструменты: context7, dribbble.com, 21st.dev
Задачи:
  1. Анализ Dribbble (5-7 референсов) → design-brief.md
  2. Выбор стека и файловой структуры
  3. Написание всего кода — компоненты, страницы, стили
  4. Адаптация паттернов с 21st.dev под дизайн-систему
  5. Обновление design-system.md при каждой итерации стилей
```

### Агент B — `qa-security`
Запускается после завершения страницы или крупного блока. Не после каждого компонента.

```
Инструменты: playwright MCP, context7
Задачи:
  1. E2E тесты через Playwright
  2. Security-аудит по полному чеклисту (см. раздел Безопасность)
  3. Адаптивность: 375 / 768 / 1280 / 1920px
  4. Lighthouse: Performance ≥ 90, Accessibility ≥ 90
  5. Фиксация в test-report.md
```

**Токен-правило:** Агент B = 1 запуск на страницу, не на компонент.

## 🚀 TEAM MODE — полная агентная система

**Когда:** 7+ страниц, бэкенд, аутентификация, команда, сложная дизайн-система, срок > 2 недель.

Агенты работают последовательно, передавая артефакты.

### Агент 1 — `design-researcher`
```
Инструменты: dribbble.com, context7
Когда: только при старте проекта или смене концепции — один раз
Задачи:
  1. Анализ Dribbble — топ 10 работ по теме
  2. Выбор 2-3 стилистических направления
  3. Фиксация: палитра, типографика, анимационный стиль, сетка
Артефакт: design-brief.md
```

### Агент 2 — `architect`
```
Инструменты: context7
Вход: design-brief.md
Задачи:
  1. Выбор стека (Next.js / Astro / Vite+React / plain HTML)
  2. Файловая структура и компонентная архитектура
  3. Роутинг, state-management, API-слой
  4. Краткие, конкретные решения без лишней философии
Артефакт: arch-decisions.md
```

### Агент 3 — `developer`
```
Инструменты: context7, 21st.dev
Вход: design-brief.md + arch-decisions.md (только они, не вся история)
Задачи:
  1. Реализация компонентов и страниц
  2. Mobile-first, семантический HTML5
  3. CSS-переменные для всех дизайн-токенов
  4. Адаптация паттернов с 21st.dev (не копипаст)
  5. Обновление design-system.md при изменении стилей
Правила кода:
  - Компоненты изолированы, нет глобальных побочных эффектов
  - Нет inline styles, нет !important без явной причины
  - Семантический HTML: <main>, <article>, <section>, <nav>
```

### Агент 4 — `qa-security` (объединённый)
```
Инструменты: playwright MCP, context7
Вход: готовый билд или страница
Когда: после завершения страницы/модуля, не после каждого коммита

Тестирование:
  — Smoke: загрузка, навигация, формы, console errors = 0
  — Адаптивность: 375 / 768 / 1280 / 1920px
  — Доступность: контрастность ≥ 4.5:1, клавиатура, alt-атрибуты
  — Производительность: LCP < 2.5s, CLS < 0.1

Security-чеклист:
  [ ] CSP настроен
  [ ] Нет API-ключей в коде и репозитории
  [ ] XSS: формы экранированы и валидированы на клиенте + сервере
  [ ] HTTPS-only, no mixed content
  [ ] npm audit — нет critical уязвимостей
  [ ] robots.txt и sitemap.xml корректны
  [ ] rate-limiting на серверных роутах (если есть)
  [ ] CORS минимален
  [ ] Нет чувствительных данных в localStorage без шифрования
  [ ] HTTP-заголовки: X-Frame-Options, Referrer-Policy, Permissions-Policy

Артефакты: test-report.md, security-report.md
```

## 🎨 Работа со стилями — 21st.dev

При каждом новом промте на стили от пользователя:

```
1. АНАЛИЗ    → Что меняется? Цвет / типографика / анимация / компонент?
2. ИЗОЛЯЦИЯ  → Найди CSS-переменную или компонент — меняй только его
3. АДАПТАЦИЯ → Паттерн с 21st.dev встраивай в дизайн-систему проекта
4. ТЕСТ      → Playwright: скриншот до/после, проверь адаптивность
5. ФИКСАЦИЯ  → Обнови design-system.md
```

**Правило экономии:** не перезапускай полный QA ради правки цвета кнопки. Playwright — только для структурных изменений.

## 🔒 Безопасность — постоянные правила (все режимы)

### Никогда
- ❌ Хранить секреты в коде (ключи, токены, пароли)
- ❌ `dangerouslySetInnerHTML` без санитизации
- ❌ Доверять пользовательскому вводу без валидации
- ❌ Коммитить `.env`
- ❌ Отключать CSP для удобства

### Всегда
- ✅ `.env.example` в репозитории (без реальных значений)
- ✅ Валидация форм на клиенте И сервере
- ✅ `npm audit` перед каждым деплоем
- ✅ HTTPS везде: ресурсы, шрифты, API
- ✅ Заголовки безопасности

```
Content-Security-Policy: default-src 'self'; ...
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

## 🛠 MCP Инструменты

### context7
Актуальная документация библиотек. Использовать когда:
- Нужен точный API конкретной версии пакета
- Настройка инструмента вызывает сомнения
- Незнакомый метод перед использованием

### playwright MCP
E2E тестирование и визуальная проверка. Использовать когда:
- Завершена страница или крупный блок
- Финальный QA перед деплоем
- Визуальная регрессия после обновления стилей
- Проверка адаптивности

**Не использовать** после каждого мелкого изменения.
